1. Introduction

Définitions et concepts de base

Bit
Unité d'information binaire (Binary unit), pouvant prendre les valeurs 0 ou 1.
Câble ethernet
Câble permettant de relier des équipements informatiques entre eux. Ce sont des câbles en paires torsadées (twisted pair) pour éviter les interférences électromagnétiques.
Routage
Technique permettant de déterminer le chemin à emprunter pour acheminer des données d'un point à un autre sur un réseau.
Forwarding
Action de guider le paquet de données vers sa destination.
Packet-switching
Technique de transmission de données où les données sont découpées en paquets qui sont envoyés séparément. Les paquets peuvent emprunter des chemins différents pour arriver à destination. Il arrive que les paquets soient perdus.
Store-and-forward (si le noeud est dispo), le paquet est stocké puis envoyé avec un délai de LR\frac{L}{R}RL​ secondes (paquet de LLL bits envoyé à un débit de RRR bps).
Queuing (si le noeud est occupé), le paquet est mis en liste d'attente puis envoyé dès que possible. Si la file d'attente est pleine, les paquets entrants sont ignorés et perdus.
Circuit-switching
Alternative au Packet-switching, les chemins sont établis à l'avance donc les noeuds sont moins sollicités. Peu utilisé sur Internet car un système ne peut avoir qu'un seul interlocuteur à la fois, ce qui augmente énormément le nombre de noeuds.

Aujourd'hui, certains protocoles requièrent un transfert de données plus fiable donc on utilise les deux sur Internet : le circuit-switching pour quelques routes privilégiées et le packet-switching pour le reste.

Frequency Division Multiplexing (FDM)
Technique de multiplexage qui consiste à diviser la bande passante d'un même support physique en plusieurs canaux de fréquences distinctes. Chaque canal peut transmettre au débit maximal qui lui est alloué.
Time Division Multiplexing (TDM)
Technique de multiplexage qui consiste à diviser le temps d'accès à un support physique en plusieurs intervalles de temps entrelacés. Chaque intervalle est alloué à un canal différent. Le débit de chaque canal dépend donc du nombre d'utilisateurs

Internet

Internet Service Provider (ISP, ou FAI en français)

Entreprise qui fournit des services d'accès à Internet.

Interconnexion : Les ISP sont interconnectés entre eux pour permettre à leurs clients de communiquer avec les clients des autres ISP (dans la limite des accords et géorestrictions).

Politiques nationales : L'évolution du réseau est influencée par l'économie et les politiques nationales.
Exemple : la connexion Europe-USA est plus rapide que Suède-Russie

Comment connecter les ISP entre eux ?

  • Connecter chaque réseau à tous les autres :
    Complexité en O(n2)O(n^2)O(n2) → trop complexe donc impossible.
  • Connecter chaque réseau à un ISP global :
    Complexe sur le plan politique.
  • Conecter chaque réseau à des réseaux régionaux :
    Ils sont reliés entre eux par des IXP (Internet Exchange Point). C'est ce qui est fait aujourd'hui.
    Schéma : Accès ISP → ISP Régional → IXP → Tier-1 ISP (réseau mondial)

Performances

Les pertes de données se font lorsque la file d'attente est pleine, et la file d'attente se remplit lorsque le nombre de paquets en entrée dépasse les capacités de sortie du noeud

Temps d'attente (queueing delay)
Temps passé par un paquet dans la file d'attente, dépend de la capacité de sortie du noeud.
dqueue=L×aRd_{queue} = \frac{L \times a}{R}dqueue​=RL×a​, avec LLL la taille du paquet, aaa le taux d'arrivée des paquets et RRR le débit de sortie du noeud.
Si le taux aR\frac{a}{R}Ra​ est supérieur à 1, la file d'attente est infinie.
Temps de nodal processing
Temps nécessaire pour traiter le paquet au niveau du noeud (d'où il vient, où il va, s'il y a des erreurs, etc.)
Temps de transmission
Temps nécessaire pour insérer le paquet sur le lien physique.
dtransmition=Lvtd_{transmition} = \frac{L}{v_t}dtransmition​=vt​L​, avec LLL la taille du paquet et vtv_tvt​ la vitesse de transmission du lien physique.
Temps de propagation
Temps nécessaire pour que le signal se propage d'un noeud à un autre.
dpropagation=dvpd_{propagation} = \frac{d}{v_p}dpropagation​=vp​d​, avec ddd la distance entre les deux noeuds et vpv_pvp​ la vitesse de propagation du signal (environ 2×1082 \times 10^82×108m/s dans un câble).
Bande passante
Taux de transmission maximal sur un lien de communication.
Throughput
Taux de transmission (bps) auquel les paquets sont actuellement transmis sur le réseau. Il peut être instantané ou moyen.

Sécurité du réseau

Packet Sniffing
Technique qui consiste à intercepter les paquets de données qui circulent sur un réseau (en se mettant sur le même réseau que la victime).
IP Spoofing
Technique qui consiste à usurper l'adresse IP d'un autre ordinateur pour se faire passer pour lui.
Denial of Service (DoS)
Attaque qui consiste à saturer un serveur en lui envoyant un grand nombre de requêtes. La bande passante est saturée et le serveur ne peut plus répondre aux requêtes légitimes.

Modèle OSI

Fonctionnement pour l'envoi d'un message (Encapsulage) :

  • Il est crée dans la couche application (Message : [M])
  • Il est encapsulé par la couche transport (Ajout de l'en-tête [Ht|M], l'en-tête peut être le nombre de bits, le numéro de séquence, etc.)
  • La couche réseau précise l'IP... (Message : [Hn|Ht|M])
  • La couche lien transmet le message sur le réseau (Message : [Hl|Hn|Ht|M])
  • Le lien physique transmet le message (Message : 010101101011100)

Idem dans l'autre sens pour la réception du message (Décapsulage)

S'il y a un intermédiaire :

  • Les switches lisent la couche physique et lien pour savoir où envoyer le message
  • Les routeurs lisent la couche physique, lien et réseau pour savoir où envoyer le message

En théorie, ils ne peuvent pas lire le message (mais un hacker pourrait modifier son switch pour le faire)

2. Historique d'Internet

1961-1972 : Premiers principes de packet-switching

  • 1961 - Kleinrock publie le premier papier sur la théorie des files d'attente, qui montre l'efficacité du packet-switching.
  • 1964 - Baran propose un réseau décentralisé (qui permettrait de survivre à une attaque nucléaire).
  • 1967 - ARPAnet : Premier réseau à utiliser le packet-switching, financé par le département de la défense américain. et conçu par l'Advanced Research Projects Agency (ARPA).
  • 1969 - Premier réseau opérationnel ARPAnet.
  • 1972 - Démonstration publique d'ARPAnet.
    NCP (Network Control Protocol) est le premier protocole host-host.
    Premier programme de mail.
    ARPAnet a 15 noeuds (= 15 périphériques connectés).

1972-1980 : Fonctionnement inter-réseau, nouveaux réseaux

  • 1970 : ALOHAnet (Université d'Hawaï) : Premier réseau sattelite.
  • 1974 : Cerf et Kahn publient le premier papier sur l'architecture des réseaux inter-connectés.
Principes de fonctionnement de l'Internet de Cerf et Kahn :
  • Minimalisme, autonomie
  • Ne pas garantir l'accès à tous les noeuds réseau en permanence
  • Rooting non-étatique
  • Contrôle décentralisé de la ressource
  • 1976 : Ethernet (Xerox, Intel, DEC) : Premier réseau local.
  • 1979 : ARPAnet a 200 noeuds.

1980-1990 : Nouveaux protocoles, prolifération

  • 1982 : Protocole SMTP (Simple Mail Transfer Protocol) : Premier protocole de mail.
  • 1983 : Déploiement du TCP/IP (Transmission Control Protocol/Internet Protocol). Le DNS (Domain Name System) définit les traduction name-to-IP.
  • 1985 : Protocole FTP (File Transfer Protocol) : Premier protocole de transfert de fichiers.
  • 1988 : Ajout du contrôle de congestion TCP

Nouveaux réseaux nationaux : CSnet, BITnet, NSFnet, Minitel, etc.

1990-2000 : Commercialisation, croissance

  • Débuts 90 : ARPAnet est démantelé.
  • 1991 : Tim Berners-Lee (CERN) invente le World Wide Web. Le protocole HTTP (HyperText Transfer Protocol) et le langage HTML (HyperText Markup Language) entrent dans le domaine public.
  • 1993 : Mosaic, premier navigateur web (deviendra Netscape puis Firefox).
  • Fin 90 : Commercialisation du web, messageries instantanées, partage de fichiers P2P (piratage de jeux, de musique...), importance de la sécurité, 50M d'hébergeurs, 100M d'utilisateurs, mise à jour des infrastructures pour augmenter la bande passante.

2005-Aujourd'hui : Changement d'échelle, cloud

  • Les fournisseurs de services Internet (Google, Amazon, Facebook, etc.) fournissent des services "cloud".
  • Les données ne sont plus stockées localement mais portées sur le réseau.
  • Apparition du streaming
  • 18 milliards d'appareils connectés à Internet

3. Couches Application et Transport

3.1. Couche Application

  • Protocoles : HTTP, FTP, SMTP, DNS, etc.

  • Paradigme client-serveur
    Le serveur héberge en permanence ("always-on") et a une adresse IP permanente (si ce n'est pas le cas, le serveur fait partie du dark web).
    Le client contacte et communique avec le serveur. Il peut être connecté de façon intermittente et peut avoir une adresse IP dynamique.
    Utilisé en HTTP, IMAP, FTP, etc.
  • Paradigme pair-à-pair (P2P)
    Pas de serveur permanent, les pairs communiquent directement entre eux pour se fournir eux-même un service. Les pairs peuvent être connectés de façon intermittente.
    Utilisé pour le partage de fichiers (BitTorrent), la téléphonie (Skype), etc.
  • Comment deux applications communiquent ?
    Les applications utilisent des sockets pour communiquer. Un socket est un point de communication entre deux programmes sur un réseau.
    Un socket est identifié par une adresse IP et un numéro de port.
    Port Protocole
    80 Serveur HTTP
    25 Envoi de mail SMTP
    465 Envoi de mail SMTPS (sécurisé)
    20/21 Envoi de fichier via FTP
    53 DNS
    443 HTTPS
    Un serveur a un socket pour chaque client, un client a un socket pour chaque serveur.
    Pour communiquer, il faut utiliser des couches de protocoles définis à l'avance (Type de message échangé, syntaxe (texte claire ? encrypté ? caractère de fin?), sémantique, etc.)
  • De quel transport a besoin une application ?
    Besoin d'intégrité des données
    Certaines applications requièrent un transfert 100% fiable (fichiers, transactions), alors que d'autres non (appel audio, réunion vidéo).
    Timing
    Certaines applications requièrent un court délai (internet, téléphonie, jeux en ligne), alors que d'autres non (mail, transfert de fichiers).
    Throughput
    Certaines applications nécessitent une certaine bande passante (streaming vidéo, transfert de fichiers), alors que d'autres non (applications "élastiques").
    Sécurité

3.2. Couche Transport

  • Deux protocoles majeurs : TCP et UDP

  • UDP (User Datagram Protocol)

    • Non fiable,
    • Ne vérifie pas qu'une connexion est possible avant d'envoyer les données,
    • Ne fournit pas de contrôle de flux, de congestion, de sécurité,
    • Plus rapide.
    • Très peu utilisé (éventuellement pour le streaming à très grande échelle (diffusion de la NASA) et la téléphonie par Internet)

  • TCP (Transmission Control Protocol)

    • Fiable,
    • Contrôle du flux,
    • Contrôle de la congestion,
    • S'assure avant tout que la connexion est possible,
    • Connexion point à point (émetteur, récepteur) et bidirectionnelle,
    • Ne fournit pas le service le plus rapide/sans latence, besoin d'une bande passante minimale, faible sécurité.
    • Pas de cryptage, les mots de passe traversent Internet en clair.

  • TLS (Transport Layer Security)

    • Protocole de sécurité qui encrypte les connexions TCP.
    • Utilisé dans HTTPS, SMTPS, etc.
    • Implémenté dans la couche application.

  • Utilisation des sockets

    • Création de l'illusion d'une connexion directe entre les deux applications.

    Exemple de communication TCP :

    • Serveur : Crée un socket, attend une demande de transaction.
    • Client : Crée un socket, demande une connexion au serveur en spécifiant l'adresse IP et le numéro de port.
    • Serveur : Accepte la connexion, crée un nouveau socket pour la communication avec le client.
    • Client : Reçoit l'accord du serveur et envoie le message.
    • Serveur : Reçoit le message sur son socket de connexion, envoie une réponse et ferme le socket de connexion.
    • Client : Reçoit la réponse et ferme le socket de connexion.

    (En UDP, il n'y a pas de connexion, le client envoie directement le message au serveur).

  • Structure d'un segment TCP (Page wikipédia)

    • Port source
    • Port destinataire
    • Numéro de séquence (numéro du premier octet du segment)
    • Numéro d'acquittement (numéro du prochain octet attendu)
    • Longueur de l'en-tête
    • Fanions (flags) binaires :
      • flag A (ACK) == 1 signifie que le paquet contient un acquittement
      • flag R (RST) demande un reset de la connexion
      • flag S (SYN) contient une requête de connexion
      • flag F (FIN) indique que le paquet est le dernier de la connexion
    • Checksum (vérification de l'intégrité du segment)
    • Fenêtre de réception (nombre d'octets que le destinataire est prêt à accepter, pour ne pas le surcharger, voir le contrôle de flux)
    • Pointeur d'urgence (optionnel, pour les données urgentes)
    • Options TCP (facultatives)
    • Données envoyées

  • RDT 3.0 (Reliable Data Transfer)

    • Solution trouvée pour améliorer le taux d'utilisation des liens (pour ne pas gaspiller de la bande passante).
    • Pipelining : Envoi de plusieurs paquets à la suite sans attendre l'acquittement de chacun. Le récepteur les valide tous d'affilée avant que l'émetteur puisse en envoyer d'autres.
    • Go-Back-N :
      • Consiste à envoyer x paquets à la fois. Lorsque le paquet n est acquitté, on décalle la fenêtre d'envoi à n+1 et on envoie le paquet suivant.
      • Si on reçoit l'acquittement d'un paquet qui n'est pas le premier, c'est un acquittement cumulatif qui acquitte tous les paquets avant lui.
      • Si on reçoit les paquets dans le désordre, on peut soit envoyer un acquittement pour le dernier paquet reçu (l'envoyeur aura un acquittement dupliqué = problème de transport), soit on attend un temps de garde pour laisser le temps de recevoir le paquet puis envoie un acquittement cumulatif si on le reçoit bien (un seul pour plusieurs pour réduire la charge sur réseau)

  • Définir le temps de garde TCP

    • Il doit être plus long que le RTT (Round Trip Time, temps mis par un paquet pour aller du client au serveur et revenir) pour éviter les acquittements dupliqués.
    • S'il est trop long, le débit est réduit.
    • S'il est trop court, des acquittements seront dupliqués inutilement.
    • Pour estimer le RTT, on estime la vitesse du récepteur (dépend de sa vitesse de base et de sa charge). TCP contient un mécanisme de contrôle de flow (mécanisme de rétro-action qui permet au récepteur d'indiquer à l'émetteur de ralentir s'il est surchargé).

  • Connexion client-serveur

    • 3-way handshake :

      • Les sockets client et serveur sont en écoute (LISTEN)

      SYN :

      • Le client envoie un message avec SYN=1 et Seq=x (SYNSENT)

      SYN-ACK :

      • Le serveur reçoit le message, et envoie un paquet avec SYN=1, ACK=1 et Seq=y, Acknum=x+1 (SYNRCVD)

      ACK :

      • Le client reçoit le message SYNACK(x), et envoie un paquet avec ACK=1 et Acknum=y+1 (ESTAB)
      • Le serveur reçoit le message ACK(y), et la connexion est établie (ESTAB)

  • Fermeture de connexion

    • Problème : Si le client ferme la connexion, le serveur ne le sait pas et continue d'envoyer des données.

    • Solution :

      • Le client et le serveur ferment leur connexion en envoyant un FIN chacun.
      • Le serveur envoie un ACK pour confirmer la fermeture de la connexion.
      • Le client envoie un ACK pour confirmer la fermeture de la connexion.
      • La connexion a été fermée en simultané !
  • Congestion

    La congestion se manifeste par des délais de traitement (RTT) beaucoup plus longs et par des pertes de paquets.

    Scénario simple :

    • Deux émetteurs veulent se partager un lien avec une certaine capacité R,
    • Si la somme des paquets envoyés par les deux émetteurs ne dépasse pas R, on peut continuer à envoyer,
    • Sinon, le délai augmente progressivement jusqu'à devenir infini pour chaque nouveau paquet envoyé.

    Scénario complexe :

    • Dans un réseau de plusieurs routeurs, certains commencent à saturer, le débit augmente alors progressivement dans les autres jusqu'à une chute brutale (congestion collapse),
    • La seule solution pour résoudre le problème est de redémarrer tous les routeurs du réseau/d'Internet :(
    • Aujourd'hui, les routeurs doivent réduire automatiquement leur débit en cas de problème pour éviter le congestion collapse.

    Contrôle de congestion TCP :

    • Approche AIMD (Additive Increase/Multiplicative Decrease) :
      • Additive Increase : Augmentation linéaire du débit tant que le réseau n'est pas saturé,
      • Multiplicative Decrease : Réduction du débit de moitié en cas de congestion.
      • Problème : TCP teste constamment la limite du réseau, ce qui consomme des ressources.

    • Mécanisme Slow Start :
      • Augmentation exponentielle du débit tant que le réseau n'est pas saturé (càd jusqu'au premier paquet perdu)
      • Quand on se rapproche du seuil (ssthresh), on passe en congestion avoidance (augmentation linéaire)
      • Si on perd un paquet, on recommence tout.

    • Mécanisme Fast Retransmit :
      • Le débit monte exponentiellement comme dans Slow Start, mais si on perd un paquet il redescend au seuil puis il remonte de façon linéaire.

3.3. Fonctionnement d'une application sur Internet

3.3.1. Applications WEB, HTTP

HTML (HyperText Markup Language)

Langage de balisage pour la création de pages web.

HTTP (HyperText Transfer Protocol)

Protocole de transfert de données sur le web.

Utilise le protocole TCP (port 80) pour transférer les données.

Deux versions du protocole HTTP :

  • Persistent HTTP : Le serveur garde la connexion ouverte entre plusieurs demandes de ressources.
  • Non-persistent HTTP : La connexion doit être rétablie pour chaque demande.
URL (Uniform Resource Locator)

Adresse qui permet d'identifier de façon uniforme une ressource sur Internet.

Exemple : Dans l'URL : www.quelquechose.fr/fichier/ressource.pdf,

  • "www.quelquechose.fr" est le serveur web
  • "/fichier/ressource.pdf" est la ressource
Message de requête HTTP
  • Types de requêtes :
    • GET : Envoie des données au serveur via l'URL.
    • POST : Envoie des données au serveur via le corps du message HTTP.
    • HEAD : Demande uniquement les données d'en-tête.
    • PUT : Upload un nouvel objet sur le serveur.
  • Codes de statut HTTP :
    • 200 : OK
    • 301 : Le fichier n'existe plus
    • 400 : Requête mal formulée (GET sans URL par exemple)
    • 404 : Fichier introuvable
    • 505 : Version HTTP non supportée
    • 418 : I'm a teapot :)

Sur linux, on peut tester une connexion HTTP avec la commande nc (netcat) :

% nc -c -v www.polytech.univ-smb.fr 80
% GET / HTTP/1.1 Host: www.polytech.univ-smb.fr
Cookies

Utilisés pour garder la trace des connexions précédentes

Le fichier cookie est stocké sur le client mais le serveur sait reconnaitre l'identifiant et le recoller à des informations client

Les cookies peuvent servir pour l'authentification, les paniers, les recommendations, garder un état dans la communication entre un client et un serveur

Les tracking cookies peuvent être accédés par d'autres sites (facebook peut voir qu'on est allé sur un site pour mettre des pubs (mais les navigateurs le bloquent normalement))

Cache web

Stockage temporaire des pages web pour accélérer leur affichage.

Si la ressource demandée est dans le cache, le navigateur l'envoie directement au client. Sinon, il la demande au serveur et la stocke dans le cache.

3.3.2. Applications de messagerie, SMTP, IMAP

Trois compostantes :

  • Utilisateur (client) : Celui qui se connecte au serveur de mail (Outlook, iPhone mail, pas les webmails du coup).
  • Serveurs de mail
  • Protocoles :

    • SMTP (Simple Mail Transfer Protocol) :

      • Vieux protocole,
      • Besoin de fiabilité (le mail doit arrivé comme il est parti),
      • Transfert direct, de serveur (qui agit comme un client) à serveur destinataire,
      • Trois étapes (handshaking, transfer, closure)
      • HTTP = client pull, SMTP = serveur push
      • Contenu du mail : en-tête (de, à, sujet, date) et corps, tout en ASCII et réponses en codes de statut.

    • IMAP (Internet Message Access Protocol) :

      • Permet l'extraction des mails depuis le serveur.

    • HTTP :

      • Utilisé pour les webmails (Gmail, Yahoo, universitaire, etc.)
      • Permet de lire les mails depuis le serveur.

3.3.3. Domain Name System (DNS)

  • Permet de résoudre une adresse (convertir son nom d'hôte en adresse IP).
  • Il n'y a pas une résolution par connexion à un site Internet, le navigateur garde en cache les adresses IP des sites fréquemment visités.
  • Pour identifier une adresse sur le réseau, on utilise l'adresse IPv4 (32 bits) ou IPv6 (128 bits).
  • Les noms de domaine principaux (.fr, .uk, .tv...) sont gérés par l'ICANN (Internet Corporation for Assigned Names and Numbers).

Pourquoi ne pas utiliser un DNS centralisé ?

  • Un seul point pour faire tomber Internet,
  • Beaucoup trop de trafic pour un seul serveur,
  • Difficile à maintenir.

Importance des DNS

  • Actuellement plusieurs trillions de requêtes DNS par jour,
  • Besoin de réponses rapides,
  • Millions de sous-serveurs DNS pour les organisations (polytech par exemple).

3.3.4. Architecture peer-to-peer (P2P)

Exemples : BitTorrent pour les fichiers, Skype pour le VoIP

  • Les pairs communiquent directement entre eux pour se fournir eux-même un service.
  • Les pairs peuvent être connectés de façon intermittente.
  • Pas de serveur permanent.
  • Plus il y a de pairs qui partagent un fichier, plus il est rapide à télécharger.
  • Sur BitTorrent, un fichier est divisé en segments de 256kb et les pairs s'échangent les segments.

3.3.5. Streaming vidéo

Netflix, Prime Vidéo, YouTube = 80% du trafic des ISP résidentiels

4. Couche Réseau

La couche réseau s'exécute sur chacune des machines sur le réseau
Un routeur est le système qui décide quelle machine doit recevoir le paquet en fonction de l'en-tête IP,

Fonctionnalités de la couche réseau
Forwarding : Décider de la sortie d'un paquet reçu en entrée.
Routing : Déterminer la route prise par un paquet de la source à la destination.
Data Plane : Partie du routeur qui s'exécute localement pour déterminer comment le datagramme doit être transmis du port d'entrée du routeur au port de sortie.
Control Plane : Partie du routeur qui s'exécute sur le réseau pour déterminer la route que le datagramme doit prendre.
Table de commutation/de routage : Table qui contient les adresses IP des machines connectées au routeur et les ports de sortie correspondants. Elle est mise à jour par le protocole de routage et chaque routeur a sa propre table.

  • Modèle de service réseau :

    • Garantie d'intégrité, de conservation de l'ordre des paquets, de délai faible...
    • Internet (IP) → Modèle "Best Effort", aucune garantie de délai, de livraison et de conservation de l'ordre.
    • Asynchronous Transfer Mode (ATM) → Modèle Constant Bit Rate (CBR), garantie de délai, d'intégrité et de conservation de l'ordre.
    • Le modèle "Best Effort" est le plus utilisé car il est plus simple et plus rapide (il faut uniquement gérer la bande passante).

  • Routeur :

    • Utilise les signaux pour transporter l'information (Modulation = adaptation d'une onde pour qu'elle transporte un signal)

    • La première fonction de la carte de réception/émission est de démoduler le signal pour lire le message,

    • Un composant gère la mémoire du port, pour traiter l'information d'où aiguiller le message,

    • Entre les ports, il y a un switch haute-vitesse

    • Le tout (+ la mémoire) est controlé par un rooting processor du routeur,

    • Le controle du routeur ne s'exécute que quand il y a un changement de rootage (quand il y a un changement sur le réseau) en quelques milisecondes,

    • Le transfert de données s'opère en quelques nanosecondes,

    • Schéma du port d'input :

          | couche physique : reçoit les bits
    | couche lien : Ethernet 
    V Switching décentralisé : utilise les headers, puis envoie au port de sortie

    • Le routage avec une autre adresse IP se fait avec le "longest prefix match" (on envoie le message vers l'adresse IP avec le plus de premiers caractères qui correspondent aux premiers caractères de l'adresse IP de destination)

    • Le problème, c'est de faire ça très rapidement (car les données mettent seulement 1ns à être transferées)

    • Comment envoyer un paquet du port d'entrée au port de sortie ?

      • Mémoire partagée : Les paquets sont stockés dans une mémoire partagée, le routeur lit les paquets et les copie directement dans le port de sortie (jusqu'à quelques Gpbs)
      • Bus : Utilisation d'un bus pour transposer les paquets du port d'entrée au port de sortie (jusqu'à quelques dizaines de Gpbs)
      • Matrice d'interconnexions : Ensemble de liens d'interconnexion qui construisent une matrice pour faire des transferts en parallèle, mais il faut que les paquets envoyés en parallèle aient des chemins différents.

  • Protocole IP :

    • Format datagramme IP : principalement des entetes,

    • Intérêt à envoyer des longs messages car les en-têtes sont envoyés à chaque fois,

    • Interface : connexion entre la machine et un routeur, une machine peut avoir plusieurs interfaces,

    • Un réseau local donne des adresses IP locales aux machines,

    • Adresse CIDR = Adresse de la forme a.b.c.d/x avec x la longueur du masque,

    • Pour avoir une adresse IP : au niveau local c'est le DHCP. Au niveau continental c'est le RIPE pour l'europe.

  • Routage :
    Objectif : Déterminer les "bons" chemins entre un émetteur et un récepteur (qui minimisent un critère de "cout" (temps, distance, énergie consommée...))

    Approche centralisée
    Un système décide pour tous les routages, il a toutes les informations du réseau.donc il peut mieux choisir les routes.
    Approche décentralisée
    Chaque système construit ses propres tables de routage et décide de ses routes. Réseaux de plusieurs routeurs.

    Algorithmes de routage :

    • Global/Centralisé : Tous les routeurs ont une topologie complète du réseau (très couteux).

    • Décentralisé : Processus itératif où chaque routeur échange des informations avec ses voisins pour construire sa table de routage (moins couteux mais plus complexe)

    • Statique : Les routes sont déterminées à l'avance et ne changent pas (très rapide mais pas très pratique).

    • Dynamique : Les routes sont déterminées en temps réel en fonction de la charge du réseau (plus lent mais plus efficace).

    Calcul des distances entre les routeurs :

    • Algorithme de Dijkstra (link-state routing protocol) :

      • Centralisé : Compute les chemins les moins couteux depuis un noeud, itératif (au bout de k itérations, on connait k chemins)
      • Linéaire : Chaque noeud est visité une fois, on ne revient pas en arrière.
      • Complexité : O(n2)O(n^2)O(n2), réduit à O(nlog⁡n)O(n \log n)O(nlogn) en implémentant un arbre binaire rouge et noir (Red-Black Tree).

    • Algorithme de Bellman-Ford (distance vector routing protocol) :

      • Calcule les plus courts chemins en demandant à ses voisins les distances, puis en les mettant à jour, de manière itérative.
      • Les tables de routage sont demandées régulièrement pour être mises à jour.
      • La propagation de l'algorithme est convergente, il est prouvé que le chemin obtenu au final sera le plus court et chaque noeud n'est computé qu'une seule fois.
      • Algorithme très robuste mais qui peut prendre du temps (Quand une connexion Internet saute, il peut se passer plusieurs minutes avant que le reste du réseau s'en rende compte)

    Systèmes autonomes (AS) :

    • Un système autonome est un ensemble de routeurs qui sont sous le contrôle d'une seule entité administrative (typiquement un FAI).
    • Internet est composé de nombreux systèmes autonomes qui sont interconnectés.
    • Les routeurs connectés à l'exérieur de l'AS sont les seuls à avoir des communications hors du domaine.
    • Le routage sur Internet consiste à prendre les paquets sur Internet, les envoyer à la gateway de l'AS, puis les router à l'intérieur de l'AS jusqu'à la destination.

    Border Gateway Protocol (BGP) :

    • C'est la connexion interdomaine de facto. Ce protocole permet à une AS (un routeur) d'informer ses voisins et de signaler qu'il est prêt à accepter des paquets pour certaines destinations (dans les faits, il donne le masque d'adresses accessibles à travers lui).

    • Messages BGP :

      • OPEN : Ouverture de la connexion, échange des paramètres de routage (numéros d'AS, ID de routeurs, capacités des pairs...).
      • UPDATE : Annonce des routes, retrait des routes, mise à jour des routes.
      • KEEPALIVE : Maintient la connexion ouverte (Envoyé toutes les 30 secondes par défaut, et un délai de 90 secondes sans UPDATE ou KEEPALIVE ferme la connexion).
      • NOTIFICATION : Fermeture de la connexion, erreur de routage, etc.

    • Attributs BGP :

      • AS-PATH : Liste des AS traversés par le paquet.
      • NEXT-HOP : Adresse IP du prochain routeur.
      • LOCAL-PREF : Préférence locale pour une route.
      • MED : Multi-Exit Discriminator, utilisé pour choisir la sortie d'un AS.
      • COMMUNITIES : Liste d'AS qui ont des politiques de routage similaires.
      • ORIGIN : Origine de la route (IGP, EGP, INCOMPLETE).
