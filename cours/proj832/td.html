TD1 - Aiguilleur de requêtes

Cet exercice a pour but d'étudier dans le cadre d'un environnement de programmation répartie un Aiguilleur de requêtes. Celui-ci permet de répartir la charge de traitements entre plusieurs Machines identiques (i.e. fournissant les mêmes services). Les Utilisateurs, l'Aiguilleur et les Machines sont des objets RMI.
Les Utilisateurs soumettent des requêtes à l'Aiguilleur. Pour chaque requête, l'Aiguilleur choisit de la rediriger à une Machine et une seule. La Machine choisie traite la requête, fournit le résultat à l'Aiguilleur qui le retransmet à l'Utilisateur. Pendant le temps de traitement de la requête, l'Aiguilleur peut aiguiller une nouvelle requête vers une autre Machine ou vers la même. L'Aiguilleur est donc un objet concurrent. Le degré de concurrence des Machines est étudié à partir de la question 9.

Question 2

Chaque objet Machine fournit deux services :

  • un service dit lecture qui à partir d'un nom de fichier (une chaîne de caractères) fourni en entrée, retourne les données (un tableau de taille non déterminée d'octets) correspondant au contenu du fichier
  • un service dit ecriture qui à partir d'un nom de fichier (une chaîne de caractères) et de données (un tableau de taille non déterminée d'octets) fournis en entrée, écrit les données dans le fichier. Ce service retourne un booléen qui vaut vrai en cas de succès de l'opération d'écriture.

On suppose dans un premier temps que tous les objets Machines ont accès aux mêmes fichiers et que l'écriture dans un fichier est répercutée de façon immédiate et automatique sur toutes les Machines. Donner en Java RMI la définition de l'interface Machine contenant ces 2 services.

import java.rmi.Remote;
import java.rmi.RemoteException;

public interface Machine extends Remote {
    public byte[] read(String filename) throws RemoteException;
    public boolean write(String filename, byte[] data) throws RemoteException;
}

Question 3

Sans anticiper sur les questions suivantes et sans en donner la définition explicite, expliquer à partir des indications fournies jusqu'à présent, quelle doit être l'interface de l'Aiguilleur.

Un aiguilleur a la même interface que les machines !

Question 4

On souhaite maintenant que des objets Machines puissent être ajoutés et retirés en cours d'exécution au "pool" d'objets géré par l'Aiguilleur.
Quelle(s) méthode(s) faut-il définir pour prendre en compte ces fonctionnalités ? Pour quel(s) objet(s) ? Définir en Java RMI, l'interface Controle, réunissant ces fonctionnalités. Donner en français la signification précise de chaque élément (méthodes, paramètres, etc.) défini.

import java.rmi.Remote;
import java.rmi.RemoteException;

public interface Controle extends Remote {
    public boolean addMachine(Machine machine) throws RemoteException;
    public boolean rmMachine(Machine machine) throws RemoteException;
}

Question 5

Proposer en français ou en pseudo code un algorithme, le plus simple possible, pour effectuer la répartition de charge. L'algorithme doit répartir équitablement les requêtes entre toutes les Machines sans tenir compte du fait que certaines requêtes peuvent être plus longues que d'autres.

L'algorithme le plus simple pour réparir la charge c'est le round robin (tourniquet en français). C'est un algorithme de répartition de charge qui consiste à attribuer les requêtes aux machines de manière cyclique sans tenir compte de la charge de chaque machine.

Question 6

On souhaite maintenant prendre en compte les charges des Machines. Périodiquement les Machines informent l'Aiguilleur de leur niveau de charge. Celui-ci est représenté par un entier donnant le nombre de requête en attente sur la Machine.
Quelle(s) méthode(s) faut-il définir pour prendre en compte cette fonctionnalité ? Pour quel(s) objet(s) ? Définir en Java RMI, l'interface Notification comprenant cette fonctionnalité.

On veut donc rajouter une interface pour que les machines informent l'aiguilleur de leur charge.

import java.rmi.Remote;
import java.rmi.RemoteException;

public interface Notification extends Remote {
    public boolean notifyLoad(Machine machine, int load) throws RemoteException;
}

Question 7

Finalement, comment construit-on l'interface complète de l'Aiguilleur prenant en compte l'ensemble des fonctionnalités le concernant ?

On peut construire l'interface complète de l'aiguilleur en étendant seulement les interfaces Machine, Controle et Notification (c'est beau java ptn).

public interface Aiguilleur extends Machine, Controle, Notification {}

Question 9

On considère maintenant que lors d'une opération d'écriture, la mise à jour du fichier sur toutes les Machines est à la charge de la Machine qui traite la requête d'écriture.
Faut-il ajouter des définitions aux interfaces précédentes pour prendre en compte ce cas ? Si oui, le(s)quelle(s) ? Si non, pourquoi ?

Il faut au minimum ajouter un booléen pour savoir si une écriture provient d'un utilisateur ou si c'est une mise à jour (propagation) de la part d'une autre machine.

Question 10

On s'intéresse maintenant aux incohérences qui peuvent survenir lorsque l'Aiguilleur répartit successivement deux requêtes d'écriture pour un même fichier sur deux Machines différentes.
Expliquer en quoi consiste cette incohérence. Proposer pour gérer ce problème au niveau de l'Aiguilleur, une solution simple qui ne modifie aucune interface.

Scénario :

  • Si une machine M1 reçoit une requête de mise à jour a sur un fichier, M1 doit mettre à jour le fichier et propager a aux autres machines.
  • La machine M3 reçoit une requête b pour le même fichier d'un autre utilisateur, le met à jour et le propage
  • M1 fera donc a puis b
  • M3 fera b puis a
  • M2 fera dans un ordre incertain

On peut avoir des ordres de mises-à-jour différents d'une machine à l'autre, donc le contenu des fichiers sera différent !

Solution :

  • Au niveau de l'aiguilleur, on a une liste des fichiers en cours de modification et on bloque les écritures concurrentes.

Question 11

L'Aiguilleur étant un point de passage obligé et de ce fait un goulet d'étranglement pour les requêtes et les réponses, proposer une solution n'introduisant pas de nouvelles entités pour alléger sa charge, notamment en ce qui concerne la gestion des réponses.

On peut modifier la méthode read de l'interface Machine pour qu'elle prenne en paramètre l'utilisateur qui fait la requête.

public byte[] read(String filename, Utilisateur user) throws RemoteException;

Bon en vrai c'est pas trop possible car la solution de la Q11 est incompatible avec la Q10 !

Question 12

Expliquer sans forcément les définir explicitement quelle(s) conséquence(s) cela entraîne sur les interfaces.

Pour cela il faut définir une interface Utilisateur (qui est ici un genre de serveur qu'on peut appeler pour récupérer la donnée) :

import java.rmi.Remote;
import java.rmi.RemoteException;

public interface Utilisateur extends Remote {
    public void receive(byte[] data) throws RemoteException;
}

Question 13

On peut :

  • Aléatoirement
